## 一、技术架构总览

### 1.1 架构分层

```
┌──────────────────────────────────────────────────────────────┐
│                    交互层 (Interaction Layer)                 │
│  FastAPI REST │ MCP Gateway │ A2A Protocol │ WebSocket       │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                 编排层 (Orchestration Layer)                  │
│  12-Node PredictionGraph State Machine                        │
│  parse_intent → evidence → event_encode → difficulty →       │
│  select_team → run_skills → quality_gate → debate →          │
│  fuse → calibrate → hitl_gate → publish                      │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                  领域核心层 (Domain Core)                      │
│  Evidence Blackboard │ ForecastClaim Schema │ Event Encoder  │
│  Bayesian Fusion │ Conformal Calibration │ Thompson Sampling │
└──────────────────────────┬───────────────────────────────────┘
                           │
┌──────────────────────────▼───────────────────────────────────┐
│                   数据层 (Data Layer)                         │
│  PostgreSQL │ Redis Cache │ NL2SQL │ External APIs           │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 技术选型

| 层 | 技术 | 选型理由 |
|----|------|---------|
| Web 框架 | FastAPI | 高性能异步、自动 OpenAPI 文档、Pydantic 集成 |
| 编排引擎 | 自研 PredictionGraph（兼容 LangGraph 接口） | MVP 阶段降低依赖，远期迁移 LangGraph |
| 数据库 | PostgreSQL 16 + pgvector | Checkpoint 持久化、多租户隔离 |
| 缓存 | Redis 7 | Evidence 缓存、限流计数器 |
| LLM | Claude (Anthropic API) | 意图解析、事件编码、Debate |
| 统计计算 | NumPy / SciPy / scikit-learn | 贝叶斯融合、共形校准、Thompson Sampling |
| 容器化 | Docker Compose | 本地开发 + 轻量部署 |
| 可观测 | 内置 Tracer + Ledger | 全链路追踪，远期接 Langfuse |

### 1.3 Token 成本设计原则

核心法则：**LLM 只做语义理解和异常推理，数值计算/统计融合/比例缩放/查表运算全部走代码**。

| 模式 | Token 消耗 | 触发条件 |
|------|:---:|---------|
| 快速模式（日常平峰） | ~300 | 无事件、无分歧 |
| 标准模式（有事件） | ~1100 | 有非结构化事件、无分歧 |
| 深度模式（高分歧） | ~2600 | 事件+高分歧+辩论（≤2轮） |

12 个 Skill 中 10 个为纯代码执行（0 Token），仅 D1/F1 依赖前置的 Event Encoder 结果。

## 二、核心模块技术实现

### 2.1 Evidence Blackboard（统一事实来源）

```python
class EvidenceBlackboard:
    """所有 Skill 读取数据的唯一入口。不存真实数据，只存结构化引用。"""
    
    def __init__(self, trace_id: str):
        self.trace_id = trace_id
        self._entries: dict[EvidenceType, list[EvidenceEntry]] = {}
    
    def add(self, entry: EvidenceEntry) -> None: ...
    def get(self, evidence_type: EvidenceType) -> list[EvidenceEntry]: ...
    def calculate_completeness(self, required_types: list) -> float: ...
```

**关键设计**：
- 证据类型枚举：timeseries_volume / city_dynamic_forecast / weather_alert / customer_reported_volume / diversion_record / event_vector
- 每条证据含 quality_score（完整度/新鲜度/一致性）
- 支持 TTL 自动过期

### 2.2 ForecastClaim Schema（预测主张标准）

所有 Skill 输出统一遵循：

```json
{
  "schema_version": "forecast.claim.v1",
  "trace_id": "tr_20260603_001",
  "skill_id": "B1_city_dynamic",
  "claim": {
    "type": "distribution",
    "mean": 13200,
    "p10": 11880, "p90": 14520,
    "confidence": 0.85
  },
  "evidence_refs": [...],
  "assumptions": [...],
  "risk_flags": []
}
```

### 2.3 12 节点预测 Pipeline

```
parse_intent ──→ build_evidence ──→ [event_encode]* ──→ score_difficulty
     │                                                       │
     └───────────────────────────────────────────────────────┘
                                 │
                    select_team ──→ run_skills (并行)
                                 │
                    quality_gate ──→ [debate]* ──→ fuse
                                                      │
                                              calibrate ──→ hitl_gate ──→ publish
```

*条件节点：event_encode 仅在存在非结构化事件时执行；debate 仅在分歧指数 > 0.20 时触发。

### 2.4 Bayesian Fusion（多 Skill 融合）

```python
def bayesian_fusion(claims: list[ForecastClaim], tau: float = 10.0) -> FusedForecast:
    reliability_i = exp(-MAPE_i / tau) * coverage_i * evidence_quality_i
    precision_i = reliability_i / (sigma_i ** 2)
    mu_fused = sum(precision_i * mu_i) / sum(precision_i)
    sigma_fused = sqrt(1 / sum(precision_i) + sigma_residual ** 2)
```

### 2.5 Conformal Calibration（置信区间校准）

使用历史残差分桶（site_type × event_type × horizon）的 quantile 调整预测区间，防止过度自信。

### 2.6 Thompson Sampling（自进化权重学习）

```python
class ThompsonSampler:
    def select_skills(self, bucket, candidates, n_select) -> list[tuple[str, float]]:
        # 从每个 Skill 的后验分布抽样，选 top-N
    
    def update(self, skill_id, bucket, reward):
        # T+1 实际值回流后更新后验
        posterior.mu = 0.92 * posterior.mu + 0.08 * reward
        posterior.sigma = max(0.05, posterior.sigma * 0.98)
```

### 2.7 DAAO 难度评分

```
难度分 = 0.30×(1-数据完备度) + 0.25×事件复杂度
       + 0.25×场地历史波动性 + 0.20×相似案例稀缺度
```

难度等级决定执行模式：低难度→快速模式(2-3 Skill) / 中→标准模式(3-5 Skill) / 高→深度模式(5-8 Skill+3轮Debate) / 极端→仅F6兜底。

### 2.8 四层容灾降级

| 层 | 触发条件 | 动作 |
|----|---------|------|
| Layer 1: 数据降级 | 某数据表不可用 | 过滤依赖该表的 Skill |
| Layer 2: Skill 降级 | 单个 Skill 超时/异常 | 跳过，用剩余 Skill |
| Layer 3: F6 兜底 | 所有正式 Skill 失败 | 历史中位数预测 |
| Layer 4: 业务兜底 | 所有自动化层失败 | 提示人工填值 |

# 2. 配置环境变量
