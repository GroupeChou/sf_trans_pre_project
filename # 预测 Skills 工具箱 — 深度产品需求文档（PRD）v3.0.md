## 一、产品愿景与核心理念

### 1.1 愿景宣言

> 让每一位地区业务人员，都能像搭乐高一样，用自然语言自主组合预测方法，即时完成任何小众/边缘/突发的预测需求 — 不再等待数据科学团队排期，不再受困于固化系统的能力边界。

### 1.2 核心范式转移

```
旧范式：集中式固化开发                    新范式：分布式自主组装
─────────────────────────              ─────────────────────────
数据科学家构建模型 → 排期上线          业务人员自然语言描述 → 系统即时组织
一个模型应对所有场景                    按场景自动组合预测 Skills
预测维度固化（仅到件量）                任意维度可预测
黑盒输出，不可解释                      全链路可追踪、可审计
边缘场景需排期开发                      即插即用、零等待
```

### 1.3 核心公式

```
业务自主预测 = Skills 市场（可选方法积木）
              × Planner Agent（智能选方法+排执行）
              × 场景自适应（维度/目标/方法差异化）
              × 人机协同（可解释+可调整+可沉淀）
```

### 1.4 产品定位三角

```
              智能化（自动选方法、智能路由、误差自校准）
              /\
             /  \
            /    \
           /      \
          /________\
    自主化          个性化
（业务自组织）    （场景千人千面）
```

## 二、深层业务需求拆解

### 2.1 为什么需要"自组织预测"？

调研揭示了三个不可逾越的鸿沟：

| 鸿沟 | 表现 | 根因 |
|------|------|------|
| **场景鸿沟** | 大客户临时促销（吴江3万件偏差）、功能变更（黄田4次/4月）、极端天气（沙田晚到6-7小时）、兜底场地溢出（金山复工偏差13万） | 模型无法自动感知外部动态信息，且场景不可穷举 |
| **信任鸿沟** | 模型T+1D准确率95.4% 仅优于人工93.2% 的2.2个百分点，但业务仍然不敢用 | 黑盒输出 + 一次重大失误（偏差13万）摧毁信任；业务需要可解释、可调整 |
| **能力鸿沟** | 全网1000+场地，15套预测方案，但经验"在脑子里"，人员流动即流失 | 人工经验有效但未标准化；不同场地口径不统一 |

**结论**：不是模型不够好，而是集中式开发的响应速度（排期→开发→上线）永远无法追上碎片化需求的出现速度。唯一的解法是**把预测能力下放给业务端**。

### 2.2 需求层次模型（L1-L5）

```
┌─────────────────────────────────────────────────────────────────┐
│ L5: 方法论沉淀层 — 保存/复用/分享成功的预测方案组合                  │
│     "上次台风天的方法这次还能用"                                    │
├─────────────────────────────────────────────────────────────────┤
│ L4: 自主进化层 — 从历史误差中自动学习，优化方法组合权重               │
│     "系统越用越懂这个场地的规律"                                     │
├─────────────────────────────────────────────────────────────────┤
│ L3: 自主组合层 — 业务人员描述需求，系统自动选择+编排+执行方法          │
│     "我说要预测什么，系统自己想办法"                                  │
├─────────────────────────────────────────────────────────────────┤
│ L2: 方法市场层 — 所有预测方法标准化为 Skill，可发现、可组合、可插拔   │
│     "像 App Store 一样选预测方法"                                   │
├─────────────────────────────────────────────────────────────────┤
│ L1: 基础设施层 — 多 Agent 协作、数据通道、安全隔离、全链路可观测       │
│     "底层稳定可靠，业务只管用"                                       │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 用户角色与核心诉求

| 用户角色 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| **场地运营（主要用户）** | 每天下午4点做明天的人力排布 | "给我明天白班的到件量，按库区分，考虑那个大客户明天搞直播" |
| **区域经理** | 每周/大促前看区域内场地风险 | "哪些场地预测分歧大？哪些数据不全？提前调配资源" |
| **预测管理员** | 维护方法模板、注册新 Skill、调优权重 | "吴江的客户摸底方法在金山也能用吗？怎么调参数？" |
| **数据科学家（新增角色）** | 开发新的预测 Skill 并发布到市场 | "我刚写的 Prophet 促销调整 Skill，怎么让业务用起来？" |

### 2.4 场景碎片化的量化分析

调研识别出 **10 类差异化预测场景**，每类需要不同的方法组合：

| 场景 | 预测目标 | 差异化维度 | 适用方法 | 使用频率 | 典型触发条件 |
|------|---------|-----------|---------|---------|------------|
| 日常到件量 | 场地日总量 | 无特殊 | M1 + F6 + 指数平滑 | 每天 | 平峰期、无事件 |
| 班次预测 | 班次级 | 班次拆分 | M1(shift) + A2 | 每天 | IE 排班 |
| 库区分预测 | 库区级 | 库区拆分 | A1(库区占比) + B1 | 每天 | 多库区场地 |
| 大促峰值 | 促销日峰值 | 事件放大 | ARIMA×event + M4 + F4 | 每月1-2次 | 618/双11/年货节 |
| 恶劣天气 | 天气降幅 | 事件缩减 | 历史天气类比 + F4 | 偶发 | 台风/暴雪/大雾 |
| 新场地冷启动 | 无历史场地 | 空间类比 | 同城市类比 + C5映射 | 偶发 | 新场地开业 |
| 路由变更 | 线路调整影响 | 拓扑变化 | M3(在途重分配) + G2 | 偶发 | 场地关停/线路调整 |
| 客户结构变化 | 大客户增减 | 客户维度 | M4(摸底) + D1 | 每周 | 大客户入驻/退出 |
| 倒货/溢出管理 | 规划外件量 | 外部冲击 | G2 + H1(交叉验证) | 每天(兜底场地) | 周边场地关班/爆仓 |
| 多场地对比 | 横向对比 | 空间对比 | 归一化 + 排名 | 每周 | 区域经理巡检 |

### 2.5 业务人员的真实工作流（现状 vs 目标）

```
【现状】
17:00 打开 Excel → 手动从3个系统拷贝数据 → 打电话问大客户今天出货量 
→ 丰声群问有没有倒货 → 凭经验修正数字 → 发给 IE 排班
耗时：30-60 分钟/天，全部依赖个人经验

【目标】
钉钉输入："预测明天金山到件量，大客户得物明天直播，另外昨天有台风"
→ 系统自动选方法、拉数据、并行计算 → 10秒返回预测卡片
→ 业务看一眼，点"确认"或"调整" → 自动推送给 IE
耗时：1-3 分钟/天，可解释、可审计、可复用
```

## 二-A、业务自主组装：深层需求与产品逻辑

### 2-A.1 为什么 Planner 自动选择不够？

v3.0 大量笔墨在 Planner Agent 的智能路由上——用户描述需求，系统自动选 Skills。但这只解决了一半问题：

| 场景类型 | Planner 自动选择 | 业务需要自主组装 |
|---------|:---:|:---:|
| 日常到件量预测 | ✅ "预测明天件量" | — |
| 大促峰值 | ✅ "618预测" | — |
| **新客户类型**（如社区团购自提点） | ❌ 无历史数据，无匹配 Skill | ✅ 需要业务人员手动组合：`M1(基期) + 自定义调整系数` |
| **场地特有组合**（如"金山兜底 + 得物直播 + 浦江倒货影响"） | ⚠️ Planner 可能选错组合 | ✅ 业务人员最了解本地场景的组合规律 |
| **方法实验**（如"我想试试用 Prophet + 大客户比例，不用同环比"） | ❌ Planner 按评分选，不支持业务探索 | ✅ 业务需要有方法"沙盒"自由组合对比 |
| **方法教学**（如"上次台风天老张用的什么方法？"） | ❌ Planner 是黑盒决策 | ✅ 业务需要能看到、学习、复用他人的方法组合 |

**结论**：Planner 自动选择是"推荐引擎"，自主组装是"创作工具"。两者互补而非替代。

### 2-A.2 业务自主组装的产品逻辑

```
┌──────────────────────────────────────────────────────────────────┐
│                    业务自主组装 = 三阶递进                         │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  Level 1: 选模板（一键调用）                                       │
│  "按618模板预测" → 系统加载模板的 Skills 组合 + 默认参数            │
│  适用：高频、标准化的场景                                          │
│                                                                   │
│  Level 2: 调参数（模板基础上微调）                                  │
│  "用618模板，但把客户直播系数从20%调到50%"                          │
│  适用：模板框架正确但参数需本地化                                   │
│                                                                   │
│  Level 3: 搭积木（自由组合 Skills）  ← 🆕 v3.1 核心增强            │
│  业务人员手动拖拽组合 Skills：                                      │
│  框架: [F1 集货+散货 ▾]  方法: [M1 同环比 ▾] [M4 客户摸底 ▾]      │
│  维度: [库区 ▾] [班次 ▾]  校验: [V1 基期对比 ▾]                    │
│  后处理: [P1 加权中位数 ▾]                                         │
│  适用：小众/边缘/实验性场景                                         │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 2-A.3 自主组装 vs Planner 自动选择的差异

| 维度 | Planner 自动选择 | 业务自主组装 |
|------|-----------------|-------------|
| **触发方式** | 自然语言描述 | 手动选择/拖拽组件 |
| **方法范围** | Planner 评分选出的 Top-K Skills | 所有已发布的 Skills（不受评分限制） |
| **组合自由度** | 系统决定执行拓扑 | 业务手动指定串联/并联关系 |
| **参数控制** | 系统使用默认参数 | 业务可逐个参数调整 |
| **可解释性** | "系统选的" | "我自己搭的，我知道为什么" |
| **适用场景** | 高频、标准化 | 低频、小众、实验性 |
| **失败责任** | 系统负责 | 业务负责（但系统保留审计记录） |
| **学习效应** | 无 | 业务在组装中理解方法差异 |

## 三-A、在线临时 Skill 生成（Prompt-to-Skill）

### 3-A.1 为什么需要临时 Skill？

调研揭示了大量的"一次性"场景——这些场景的出现频率太低，不值得开发一个正式 Skill，但确实需要预测：

| 临时场景 | 频率 | 为什么不能固化为 Skill |
|---------|:---:|---------------------|
| "某新平台（如Temu）突然大量出货" | 首次出现 | 无历史数据，无现成方法 |
| "隔壁城市封路，车辆全部绕行到我们场地" | 偶发 | 路径变更无规律 |
| "新开了一个冷链库区，和常温件规律完全不同" | 首次 | 库区类型新增 |
| "隔壁场地关班，件量临时分流过来，持续3天" | 偶发 | 临时性，3天后恢复 |

这些场景需要**即时生成一个临时预测方法**，而不是等数据科学家开发新 Skill。

### 4.2 三维评分公式

```
Skill 综合分 = 0.35 × 场景匹配度 + 0.30 × 数据完备度 + 0.35 × 优先级

其中：
- 场景匹配度 = f(事件标签, 推荐场景)
  - 无事件 → 基准分 0.80
  - 事件完全匹配 → 1.00
  - 事件部分匹配 → 0.50 + 0.50 × (匹配数/总数)
  
- 数据完备度 = 必需数据可用数 / 必需数据总数
  - 任一必需数据不可用 → 该 Skill 被过滤（不参与评分）
  
- 优先级 = Skill 固有优先级 / 100
  - M1=0.90, M2=0.80, M3=0.75, M4=0.70, F4=0.60, F6=0.10
```

### 4.3 执行模式自动选择

系统根据场景复杂度自动选择执行模式：

| 执行模式 | 触发条件 | Skill 数量 | 是否协商 | 耗时目标 |
|---------|---------|-----------|---------|---------|
| 🟢 **快速模式** | 日常平峰、无事件、数据完备度>80% | 2-3 个 | 否（跳过 Debate） | ≤30秒 |
| 🟡 **标准模式** | 有轻微事件、中等数据完备度 | 3-5 个 | 是（1轮） | ≤3分钟 |
| 🔴 **深度模式** | 大促/天气/新场地/高不确定性 | 5-8 个 | 是（最多3轮） | ≤8分钟 |
| ⚠️ **兜底模式** | 数据完备度<50% 或全部 Skill 被过滤 | F6 兜底 | 否 | ≤10秒 |

### 4.4 分歧处理与协商（Debate）

当多个 Skill 的预测结果分歧 >20%（分歧指数 > 0.20）时，自动触发协商：

```
Round 0: 初始预测
  M1 → 150,000 件
  M4 → 180,000 件 (客户直播 +20%)
  F4 → 120,000 件 (暴雨 -20%)
  分歧指数 DI = 0.32 > 0.20 → 触发协商
        │
Round 1: Supervisor 生成质询
  "F4: 请检查历史暴雨对金山的实际影响率，当前-20%是否偏保守？"
  "M4: 得物直播的件量预估依据是什么？"
        │
  Worker 重新执行（带质询上下文）
  M1 → 150,000 件 (不变)
  M4 → 165,000 件 (修正为保守估计)
  F4 → 135,000 件 (历史数据显示金山暴雨影响约-10%)
        │
  DI = 0.08 ≤ 0.20 → 共识达成 ✓
        │
Synthesizer 综合推荐: 155,000 件 [CI: 140,000-170,000]
```

**协商终止条件**：
- 分歧指数 ≤ 0.20（提前收敛）
- 连续2轮改善幅度 < 0.03（无效协商）
- 达到最大轮数3轮（强制终止）

### 4.5 结果校准（Platt Scaling）

LLM 综合推荐有"过度保守"倾向（向平均值靠拢）。引入 Platt Scaling 对输出进行极值拉伸：

```
原始推荐: 155,000 [CI: 140,000-170,000]
                      ↓ Platt Scaling（基于历史30天误差分布）
校准后:   162,000 [CI: 138,000-186,000]
          ↑ 主值上移 +4.5%       ↑ 置信区间展宽
```

## 四-A、Agent Team 自组织与自进化体系

### 4-A.1 从"被编排"到"自组织"：Agent Team 的进化方向

当前 v3.0 的 Agent 协作是**Supervisor 驱动**的：Planner 选 Skills → Supervisor 编排执行 → Debate Host 质询 → Synthesizer 综合。这是一种**中心化编排**模式——有效，但不够"智能体化"。

v3.1 设计目标：逐步从"Supervisor 编排"演进为"Agent Team 自组织"：

```
当前（v3.0）:                    目标（v3.1+）:
Supervisor 主导一切              Agent Team 自组织协同
─────────────────              ─────────────────────
Supervisor 选 Skills    →     Skills 主动"竞标"任务
Supervisor 编排执行     →     Skills 协商任务分配
Supervisor 质询离群     →     Skills 互相质疑（Peer Review）
Supervisor 综合推荐     →     Skills 集体共识（Consensus）
Platt Scaling 静态校准  →     持续学习闭环（AgentLoop）
权重人工调整           →     基于效果自动进化
```

### 4-A.2 Agent 记忆与身份（Agent Memory & Identity）

每个 Worker Agent 拥有持久化的记忆，跨越单次预测任务：

```
┌─────────────────────────────────────────────────────────────┐
│                  Agent 记忆体系                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  📋 短期记忆（单次任务内）                                     │
│  · 本次任务输入/输出                                         │
│  · 与其他 Agent 的交互历史                                   │
│  · 被质询的内容和修正过程                                     │
│                                                              │
│  📊 中期记忆（按场地维度）                                     │
│  · 该场地历史上我的预测偏差                                   │
│  · 该场地最适合的配套 Skills                                 │
│  · 该场地的数据质量特征（哪些表经常缺失）                      │
│                                                              │
│  🧠 长期记忆（跨场地学习）                                     │
│  · 方法改进：哪种事件修正系数最有效                           │
│  · 场景归纳：A类场地在B类场景下我的表现                       │
```

### 4-A.4 Peer Review 互相质询（去中心化 Debate）

当前 Debate 是 Supervisor 单向质询离群 Worker。v3.1 引入 Worker 之间的**对等质询**：

```
Round 1: 初始预测
  M1 → 150,000 件 (基于历史同环比)
  M4 → 180,000 件 (客户直播 +20%)
  F4 → 120,000 件 (暴雨 -20%)
  分歧指数 DI = 0.32

Round 2: Peer Review（去中心化）
  M1 质询 F4: "金山历史上5次暴雨的实际折损率是 -5% ~ -12%，你的 -20% 依据是什么？"
  F4 回应:   "我查看了华南地区暴雨影响率，未区分场地。接受修正为 -10%"
  
  F4 质询 M4: "得物直播的 +20% 是经验值还是客户确认值？"
  M4 回应:   "客户尚未确认，是保守估计。如果确认实际出货可能到 +35%"
  
  M4 质询 M1: "你用的是上周同日基期，但上周有促销，基期偏高"
  M1 回应:   "接受，改用两周前同日作为基期"
  
  ↓
  
Round 3: 修正后
  M1 → 145,000 (改用两周前基期)
  M4 → 175,000 (保持保守，标注±范围)
  F4 → 137,500 (修正暴雨折损率)
  DI = 0.10 → 共识达成 ✓
```

**Peer Review 的约束**：
- 每个 Worker 最多质询 2 个其他 Worker（避免组合爆炸）
- 质询必须基于数据/历史/方法论（不允许"我觉得"）
- 被质询的 Worker 可以"坚持原判断"（不允许被迫修改）
- 质询过程全量记录，供事后审计

```
    │ 对比偏差   │   │ (delta+原因) │   │ 实际发生  │
    └─────┬─────┘   └──────┬──────┘   └─────┬─────┘
          │                 │                 │
          └─────────────────┼─────────────────┘
                            │
                ┌───────────▼───────────┐
                │   自进化引擎           │
                │                       │
                │  ① 更新 Agent 权重     │
                │  ② 更新方法参数       │
                │  ③ 更新 Platt 校准    │
                │  ④ 更新场地画像       │
```

**具体进化维度**：

|---------|---------|---------|------|

## 四-C、难度感知的 Agent 动态编排（DAAO × DyLAN 理论映射）

### 4-C.1 为什么需要难度感知分配？

v3.1 的执行模式自动选择（Section 4.3）是**基于场景类型**的固定规则：日常 → 快速模式、大促 → 深度模式。但这忽略了同一场景下**不同场地/不同数据条件**的难度差异。

DAAO（Difficulty-Aware Agent Orchestration, WWW 2026）的核心洞察：**任务难度决定最优资源分配量**——简单任务少 Agent（省成本）、复杂任务多 Agent（保精度）。DyLAN（Dynamic LLM-Agent Network, ACL 2024）则提供：**查询级动态组装**——每次预测根据输入特征自动选择 Agent 子集。

```
                    DAAO 难度感知分配
                    ═══════════════

  任务难度                            Skill 分配策略
  ────────                           ──────────────
  🟢 低难度                          并行 2-3 个 Skill
  (数据完备>0.85, 无事件,            → 快速模式（跳过 Debate）
   场地历史稳定)                     → 成本最低，响应最快

  🟡 中难度                          并行 3-5 个 Skill
  (数据完备 0.50-0.85,               → 标准模式（1 轮 Debate）
   有轻微事件, 场地有波动)           → 平衡精度与成本

  🔴 高难度                          并行 5-8 个 Skill
  (数据完备<0.50, 大促/天气,         → 深度模式（最多 3 轮 Debate）
   新场地/兜底场地)                  → 精度优先，接受更高成本

  ⚫ 极端难度                        仅 F6 兜底 + 人工主导
  (数据完备<0.30,                    → 不做精准预测
   极端天气/系统故障)                → 标注不确定性，建议人工决策
```

### 4-C.2 难度评分模型

DAAO 论文的核心公式在产品中的映射：

```
场地-场景难度分 = 0.30 × (1 - 数据完备度)            ← 数据越缺越难
                + 0.25 × 事件复杂度                   ← 事件越多/越罕见越难
                + 0.25 × 场地历史波动性               ← 场地本身越不稳定越难
                + 0.20 × 相似案例稀缺度               ← 历史参考案例越少越难

其中：
- 事件复杂度 = 1.0（3个以上异类事件同时发生）
             = 0.5（1-2个同类事件）
             = 0.0（无事件）
- 场地历史波动性 = 近30天 MAPE 的归一化值
- 相似案例稀缺度 = 1 - (相似历史案例数 / 10)
```

**产品呈现**：在预测卡片上展示难度评级，帮助业务人员理解"为什么这次用了这么多方法"：

```
┌─────────────────────────────────────────┐
│ 📊 难度评级: 🔴 高 (得分 0.78)          │
│                                          │
│ 因素分解:                                │
│ · 数据完备度偏低 (0.52) — 在途数据缺失   │
│ · 事件复杂度高 — 大促+暴雨双重影响       │
│ · 场地波动性中等 (MAPE 9.2%)             │
│ · 相似案例少 (历史仅 3 次双事件叠加)     │
│                                          │
│ 因此启用 6 个 Skill 并行 + 3 轮 Debate  │
└─────────────────────────────────────────┘
```

### 4-C.3 DyLAN 查询级动态组装

DyLAN 的核心思想：**不要为所有查询预定义固定的 Agent 团队**——而是每次预测时根据输入特征动态选择 Agent 子集。

在 v3.2 中的实现：

```
当前（v3.1）:                       目标（v3.2 DyLAN 模式）:
每个场地预配置默认 Skills            每次预测动态计算哪些 Skills 最相关
───────────────────                 ──────────────────────────────
金山 → 默认: M1 + M4 + F6         金山, 日常, 无事件 → M1 + F6
                               金山, 大促, 直播 → M1 + M4 + F4_promo + S2_Prophet + F6
                               金山, 台风, 倒货 → M1 + M4 + F4_weather + G2 + V4 + F6
```

**DyLAN 动态组装流程**：

```
用户输入 → Intent Parser 提取特征向量
       │
       ├── 场地特征: [兜底型, 大客户占比高, 多库区, ...]
       ├── 日期特征: [周一, 6月, 非节假日, ...]
       ├── 事件特征: [大客户直播, 暴雨, ...]
       └── 数据特征: [C1✓, C4✓, V1✓, V4✗, ...]
              │
              ▼
      ┌──────────────────────────┐
      │ 难度评分模型              │
      │ → 难度分 = 0.78 (高)     │
      │ → 推荐 Skill 数 = 5-6    │
      └──────────┬───────────────┘
                 │
      ┌──────────▼───────────────┐
      │ DyLAN 动态选择层          │
      │ 输入: 特征向量 + 难度分   │
      │ 输出: Skill 子集 + 权重   │
      │                          │
      │ 算法: 多臂老虎机 (MAB)   │
      │ · 探索: 10% 概率尝试新组合│
      │ · 利用: 90% 概率选最优组合│
      └──────────┬───────────────┘
                 │
                 ▼
      Planner → 执行计划（含选中理由 + 置信度）
```

### 4-C.4 DAAO/DyLAN 的能力边界

| 机制 | 当前可用性 | 依赖条件 | 降级策略 |
|------|:---:|------|------|
| 难度评分 | MVP 可用 | 场地画像 + 数据完备度（已有） | 缺少场地波动数据时，波动性默认 0.5 |
| DyLAN 动态选择 | Phase 2+ | 每个 Skill 在每种场景下的 MAPE 数据 ≥ 30 条 | 降级为固定规则选择（与 v3.1 一致） |
| 探索-利用平衡 | Phase 3+ | 大量运行数据支撑 MAB 收敛 | 降级为纯利用（选历史最优） |

### 7.3 M2: Skills 市场 [P0]

**功能描述**：所有预测方法的"应用商店"，支持注册、发现、管理。

**核心功能**：
- Skill 注册：数据科学家编写 YAML 定义 + Python 实现 → 提交审核 → 发布

- Skill 详情：定义规范、效果数据、使用统计、依赖关系
- Skill 管理：启停、权重调整、版本管理、弃用标记
- 场景模板：预设组合，业务人员一键调用

### 7.4 M3: Planner 智能路由 [P0]

**功能描述**：接收用户意图后，自动选择最优 Skill 组合并编排执行。

**核心能力**：
- 三维评分自动选 Skill（场景匹配 × 数据完备 × 优先级）
- 数据依赖性自动检查（必需数据不可用→自动过滤）
- 执行拓扑自动编排（独立 Skill 并行，依赖 Skill 串联）
- 执行模式自动选择（快速/标准/深度/兜底）
- 失败自动降级（Skill 超时 → 跳过 → F6 兜底 → 仍返回结果）

### 7.5 M4: 协同决策看板 [P0]

**功能描述**：让用户看到"系统是怎么算出来的"。

**展示内容**：
- 各 Skill 预测值对比（表格+条形图）
- 每个 Skill 的推导逻辑（数据→模板→参数→计算→结果）
- 协商过程时间线（如有 Debate）
- 分歧指数变化曲线
- 最终综合推荐及置信区间

### 7.6 M5: 人工校验台 [P0]

**功能描述**：业务人员在线调整预测值。

**核心功能**：
- 三种调整方式：加减值 / 覆盖 / 按比例
- 必填调整原因（≥5字）
- 事件标签多选（可新增自定义标签）
- 二次确认（调整幅度 >10% 时）
- 调整前后对比展示
- 审计记录自动生成（谁、何时、调了多少、为什么）

### 7.7 M6-M10: 分析与治理模块 [P1-P2]

详见模块总览表，这些模块在 MVP 后逐步上线。

### 8.2 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 编排框架 | LangGraph（当前）→ AgentScope/HiClaw（远期） | 当前已验证可行；远期向企业级生态迁移 |
| Skill 执行 | Python 确定性执行 + LLM 辅助 | 预测计算不需要 LLM 参与；LLM 只做路由和综合 |
| 通信协议 | REST API（当前）→ A2A/MCP（远期） | 优先保证功能可用，再标准化 |
| 数据通道 | NL2SQL（当前手动封装）→ 统一 API | 降低数据获取门槛 |
| 会话持久化 | MemorySaver → SQLite → PostgreSQL | 逐步从开发到生产 |
| 可观测 | Langfuse（当前） | 全链路 Trace、Token 消耗、延迟监控 |

### 8.3 Skills 执行隔离

每个 Skill 在独立沙箱中执行：
- 资源限制（CPU/内存/时间）
- 网络隔离（仅允许访问指定的数据 API）
- 超时控制（默认 30 秒，大计算 Skill 可配置更长）
- 失败不传播（单个 Skill 失败不影响其他 Skill）

### 8.4 安全与合规

- 数据访问按场地/区域鉴权
- 预测结果/人工调整全量审计
- LLM 调用不传输原始敏感业务数据（仅传输聚合统计值）
- Skill 代码审核机制（新 Skill 上线前需管理员审核）

### 8-A.2 理论基石：三大前沿突破

本架构设计严格遵循 2025-2026 年 Agentic AI 领域的核心理论：

#### 理论 1: Google Agent Scaling Law（2026.01）

> 论文: *"Towards a Science of Scaling Agent Systems"* | 实验: 180种配置 × 4基准 × 3模型家族

| 核心发现 | 数据 | 对本产品的约束 |
|---------|------|--------------|
| 可并行化任务集中式协调最优 | +80.9% 性能提升 | 多 Skills 并行预测使用 Supervisor 模式 |
| 顺序型任务 Multi-Agent 有害 | -39% ~ -70% | 预测流水线不宜过长，避免串联过多 Skills |
| 工具 > 8 个开销指数放大 | β=-0.267, p<0.001 | 每个 Skill 依赖的数据表 ≤ 5 个 |
| 单 Agent > 45% 后加人收益为负 | 能力饱和点 | 当前 6 个核心 Skill 的规模合理 |
| 独立型多智能体错误放大 17.2× | vs 集中式 4.4× | 必须保留 Supervisor 校验层和 Debate 机制 |

#### 理论 2: Nexus 双分辨率时序预测（2026.05）

> 论文: *"Nexus: An Agentic Framework for Time Series Forecasting"* (arXiv:2605.14389)

**五角色架构**：
- ① 历史上下文 Agent（结构化多模态数据 → 因果时间线）
- ② **宏观推理 Agent**（自上而下 → 整体趋势/大方向）
- ③ **微观推理 Agent**（自下而上 → 短期波动/事件敏感）
- ④ 预测合成 Agent（合并宏微观视角 → 最终数值+理由）
- ⑤ 校准 Agent（回测模拟 → 批判规则 → 持续进化）

**对我们产品的映射**：Planner 路由时引入 Macro/Micro 双视角——Macro 看城市/业务区整体趋势，Micro 看场地单日/班次异常。合成时优先采纳视角一致的预测，分歧时触发 Debate。

#### 理论 3: Bridgewater AIA Forecaster（2025）

> 机构: 桥水基金 AI 实验室

| 核心发现 | 对本产品的映射 |
|---------|--------------|
| **搜索驱动协调 > 简单加权** | Debate 机制采用"发现分歧→针对性质询→重执行"而非简单平均 |
| **必须引入 Platt Scaling** | 在 Synthesizer 输出后增加 Calibrator 统计校准层 |
| **表现与人类超级预测者统计不可区分** | 验证了"AI + 人工校验"协同模式的理论上限 |

## 八-C、容灾与降级策略完整设计

> 本节回应调研专题报告和 LangGraph 技术方案中识别的系统稳定性风险。调研中黄田场地模型偏差 13 万件、沙田在途数据缺失、金山倒货无系统记录——这些场景的共性要求是：**系统在任何故障模式下都必须能返回可用结果**。

### 8-C.1 四层兜底架构

```
Layer 4: 业务兜底 — 人工填值（任何情况都能出数）
  触发: 所有自动化层失败
  ↓
Layer 3: F6 兜底 Skill — 历史中位数预测（始终运行）
  触发: 所有正式 Skill 失败或数据完备度 < 0.30
  ↓
Layer 2: Skill 级降级 — 单个 Skill 失败不传播
  触发: Skill 超时/异常 → 跳过该 Skill → 用剩余 Skill 结果
  ↓
Layer 1: 数据级降级 — 数据源不可用时减少依赖
  触发: 数据表缺失 → 过滤依赖该表的 Skill → 降低数据完备度权重
```

### 8-C.2 故障场景应对矩阵

| 故障场景 | 触发条件 | 影响范围 | 应对策略 | 用户感知 |
|---------|---------|---------|---------|---------|
| **C1 表缺失** | 核心历史数据不可用 | 🔴 全部 6 个核心 Skill 被过滤 | → 仅运行 F6 兜底；标注"数据严重缺失，预测仅供参考" | ⚠️ 红色警示：数据缺失 |
| **C4/V1 缺失** | 业务区数据不可用 | 🟡 M1 被过滤（M1 依赖 C4+V1） | → 改用 M2+M3+M4 组合；降低置信区间 | 🟡 黄色提示：部分数据缺失 |
| **V4 在途数据缺失** | 在途车辆数据不可用 | 🟡 M3 被过滤 | → M3 不参与，用 M1+M2+M4 | 🟡 黄色提示：在途数据缺失 |
| **单个 Skill 超时** | 执行超过 30 秒 | 🟢 仅该 Skill | → 跳过该 Skill；F6 始终在场 | 🟢 绿色提示：某方法跳过 |
| **多个 Skill 超时** | ≥3 个 Skill 超时 | 🔴 主值不可靠 | → 降级为仅 F6 兜底 + 标注 | ⚠️ 红色警示：系统异常 |
| **Planner 路由失败** | LLM 调用异常 | 🔴 无法选择 Skill | → 回退到固定规则（场地画像预配置的默认 Skills） | 🟡 黄色提示：使用默认方案 |
| **Debate 无法收敛** | 3 轮后 DI > 0.20 | 🟡 共识度低 | → 强制综合推荐 + 标记 requires_hitl=True | 🟡 黄色提示：建议人工审核 |
| **LLM 全部不可用** | API 故障/配额耗尽 | 🔴 全链路瘫痪 | → 纯规则模式：固定选择 M1+F6 + 跳过 Debate + 公式投票 | ⚠️ 红色警示：降级运行 |
| **并发过载** | > 100 请求/分钟 | 🟡 响应延迟 | → 请求队列排队 + 超时拒绝 + 降级为快速模式 | 🟡 黄色提示：系统繁忙 |
| **钉钉通道故障** | 钉钉 API 不可用 | 🟡 主入口不可用 | → 切换 Web 管理台 + API 直接调用 + 群消息手动转发 | 🔴 用户无法通过钉钉使用 |

## 八-D、模型基线（satis/Transformer）与 Skills 工具箱的接口规范

> 本节回应用户背景中的关键信息："生产模型为黑盒 Transformer，仅提供 model.predict(x) 接口"。调研也确认：黄田场地模型 T+1D 准确率 95.4% 优于人工 93.2%，说明 satis 模型在**可覆盖的常规场景**内表现优秀。问题在于**覆盖范围不足**而非**单点精度不够**。因此，Skills 工具箱的定位不是替代 satis，而是**补全 satis 覆盖不到的场景**。

### 8-D.1 互补架构：satis 作为"第 7 个 Worker"

```
         │ 路由决策 │  │ 基线  │  │ 工具箱  │
         └────┬────┘  └───┬───┘  └────┬────┘
              │            │            │
              │    ┌───────┴───────┐    │
              │    │  融合策略      │    │
              │    │ satis 覆盖内:  │    │
```

### 8-D.2 satis 接口适配层

satis 作为"黑盒 Worker"，通过适配层接入 Skills 市场：

```yaml
# skill: S0_satis_baseline/SKILL.yaml
skill_id: "S0_satis_baseline"
name: "satis 生产模型基线"
category: "model_baseline"
version: "1.0.0"
author: "model-team"

# ── 接口定义 ──
interface_type: "blackbox_api"          # 黑盒 API 模式（非 Python 确定性执行）
api_endpoint: "http://satis.internal/predict"
request_format: "{site_code, target_date}"
response_format: "{volume, confidence}"

# ── 数据依赖（黑盒模型自行管理数据） ──
required_data: []                      # 不需要 Skills 工具箱提供数据
min_data_completeness: 0               # 不需要数据完备度检查

# ── 预测能力 ──
predicts: "arrival_volume"
granularity: "daily"
supported_dimensions:
  - "site_total"                       # 仅支持场地总量
  # 不支持班次/库区分等维度拆分
supported_events: []                    # 不支持事件调整

# ── 场景适配 ──
recommended_scenarios:
  - "常规预测"                          # 平峰期、数据稳定、无特殊事件
  - "城市维度对比"
not_recommended_for:
  - "极端天气"                          # 黑盒模型无法感知天气
  - "大客户集中出货"                     # 黑盒模型无法感知客户动态
  - "新场地冷启动"                       # 无历史数据
  - "兜底场地倒货"                       # 溢出残差无法建模

# ── 性能特征 ──
confidence_level: 0.954                # 黄田场地实测 T+1D 准确率
avg_mape: 0.046                        # 1 - 0.954
avg_latency_ms: 200                    # API 调用延迟
max_latency_ms: 2000                   # 允许的最大延迟

# ── 执行配置 ──
execution_mode: "api_call"             # API 调用模式
timeout_ms: 5000                       # 超时 5 秒
retry: 1                               # 失败重试 1 次
fallback_on_failure: "F6"              # 失败后由 F6 兜底
```

### 8-D.3 satis 与 Skills 工具箱的融合策略

| 场景 | satis 可用? | Skills 可用? | 融合策略 | 输出标记 |
|------|:---:|:---:|------|------|
| 日常平峰 | ✅ | ✅ | **以 satis 为主值**，Skills 为辅（交叉验证）| "AI 模型 + 人工方法双重验证" |
| 日常平峰（大客户占比高） | ✅ | ✅ | satis + M4 客户摸底 → 取加权结果 | "模型基线 + 大客户修正" |
| 大促/促销 | ⚠️ 不可靠 | ✅ | **仅用 Skills**，satis 作为参考 | "大促场景，以人工方法为准" |
| 恶劣天气 | ❌ 不可用 | ✅ | **仅用 Skills** | "极端天气，模型停用" |
| 新场地 | ❌ 无数据 | ✅ | Skills 冷启动模式 | "新场地，历史数据不足" |
| 倒货/溢出 | ❌ 覆盖外 | ✅ | Skills G2 + H1 兜底管理 | "兜底场地，溢出残差管理" |
| satis 预测值异常 | ⚠️ | ✅ | **自动检测** satis 预测值是否超出历史 3σ 范围 → 异常时降权或跳过 | "模型异常，已降级处理" |
| Skills 全部失败 | ❌ | ❌ | **仅 satis** → 标注"仅模型基线" | "人工方法异常，仅模型基线" |

### 8-D.4 satis 异常检测规则

```
satis 异常检测:
  IF satis 预测值 < 历史同期 30 天均值的 50%
     OR satis 预测值 > 历史同期 30 天均值的 200%
     OR satis 预测值 = 0（可能 API 异常）
  THEN
    标记 satis 为"异常"
    降权: satis_weight = 0.1（正常为 0.5）
    触发告警: "satis 模型预测值异常，请人工确认"
    以 Skills 工具箱子集结果为主
```

### 11.3 尚未解决的关键问题（诚实边界）

#### G1：Platt Scaling 校准需要 30 天以上运行数据积累 ✅（已缓解）

v3.2 缓解：Phase 0 前 30 天使用固定校准系数（基于离线测试）；Phase 2 接入 AgentLoop 后动态校准。

#### G2：Prompt-to-Skill 的代码生成质量不可控 ✅（已缓解）

v3.2 缓解：三层保护——沙箱执行隔离、结果范围校验（与 F6 兜底值对比）、明确标记"临时方法"。新增：DAAO 难度感知可帮助判断"是否应该生成临时 Skill"——低难度场景不生成，高难度场景才触发。

#### G3：业务自主组装的学习成本 ✅（已缓解）

v3.2 缓解："一键加载模板→逐步解锁参数→自由组装"的渐进式学习路径。

#### G4：跨场地知识迁移的相似度阈值 ✅（已缓解）

v3.2 缓解：初期保守（阈值=0.85），随数据积累逐步放宽。与 DAAO 的相似案例稀缺度评分联动。

#### G5：兜底型场地的根本矛盾超出产品范围 ⚠️（持续存在）

v3.2 缓解：四层兜底架构 + 容灾策略让"不可预测"有了明确的产品边界——显式区分"可预测部分"和"溢出残差"，高峰期叠加安全余量，倒货率纳入健康度指标。

#### 🆕 G6：DyLAN MAB 探索-利用需要大量运行数据（v3.2 新增）

多臂老虎机的 ε-greedy 算法需要每个 Skill 在每种场景下至少 30 条以上 MAPE 数据才能收敛。MVP 阶段不可用。**缓解**：Phase 2 前使用固定规则（DAAO 难度评分模型 + 固定阈值），Phase 3 升级为 MAB。

#### 🆕 G7：satis 黑盒模型的"不可解释性"会传染 Skills 工具箱（v3.2 新增）

当以 satis 为主值、Skills 为辅时，最终推荐值可能被 satis 拉偏而业务无法理解原因。**缓解**：始终展示 satis 与 Skills 的分项结果；当 satis 与 Skills 分歧 > 15% 时，自动切换为以 Skills 为主。

#### 🆕 G8：竞品快速迭代可能导致差异化窗口收窄（v3.2 新增）

顺丰超脑、京东京慧等产品在持续迭代。如果它们增加"业务自助"功能，本产品的差异化优势可能缩小。**缓解**：建立季度竞品监控机制；聚焦"碎片化场景"这个竞品不愿做的细分市场。
