## 二、各层详细选型与开发指引

### 2.1 平台底座层：AgentScope-first

**选型理由**：你要的是一个**面向公众开放的 SaaS 平台**，不是边缘计算集群。AgentScope 的原生能力——Agent Service（多租户）、Message Hub（通信）、Studio（可视化管理）、MCP/A2A Gateway（开放协议）——恰好是 SaaS 平台的刚需。

**开发要点**：

| 事项 | 说明 |
|------|------|
| **Agent Service** | 租户独立命名空间，每租户一个 Agent Service 实例或逻辑隔离 |
| **工具权限** | 通过 MCP tools 注册预测工具（天气 API、历史数据查询、在途车辆查询），按租户 scope 控制 |
| **消息协议** | Skill 间通信统一使用 ForecastClaim Schema (JSON)，通过 Message Hub 路由 |
| **观测面板** | 基于 AgentScope Studio 暴露：租户级调用量、Skill 表现排名、Token 消耗、审计链路 |
| **API 网关** | AgentScope 对外暴露统一 REST/gRPC 接口，租户通过 API Key 鉴权 |

**注意事项**：
- AgentScope 的 `to_dist()` Actor 分布式能力在单 SaaS 部署中仍有用——可用于隔离不同租户的 Skill 沙箱执行环境
- 如果团队对 AgentScope 生态不熟悉，RPC/gRPC 学习曲线需要预留 1-2 周

### 2.2 编排引擎层：LangGraph StateGraph

**选型理由**：预测流程是天然的状态机——`意图解析 → 取数 → 数据质量评估 → 选 Skill → 并行执行 → 协商 → 融合 → 校准 → HITL 门禁 → 发布`——LangGraph 的 StateGraph 精确匹配这个需求，且 Checkpoint + Time Travel 直接解决了审计和回滚。

**开发要点**：

```python
# 状态定义
class ForecastState(TypedDict):
    trace_id: str
    request: dict
    intent: dict
    evidence: dict
    difficulty: float
    selected_skills: List[str]
    claims: List[ForecastClaim]
    disagreement: dict
    forecast: dict
    hitl: dict

# 12 节点状态机
graph = StateGraph(ForecastState)

graph.add_node("parse_intent", parse_intent_node)           # 🔴 LLM 意图解析
graph.add_node("build_evidence", build_evidence_node)       # 🟢 纯数据拉取
graph.add_node("event_encode", event_encode_node)          # 🔴 LLM 事件→特征（有事件时）
graph.add_node("score_difficulty", score_difficulty_node)   # 🟢 VAE 小模型推理
graph.add_node("select_team", select_team_node)             # 🟢 Thompson Sampling
graph.add_node("run_skills", run_skills_parallel_node)      # 🟢 纯代码 Skill 执行
graph.add_node("quality_gate", quality_gate_node)           # 🟢 规则校验
graph.add_node("debate", bounded_debate_node)               # 🔴 LLM 有界辩论（分歧时）
graph.add_node("fuse", bayesian_fusion_node)                # 🟢 纯数学融合
graph.add_node("calibrate", conformal_calibration_node)     # 🟢 纯统计校准
graph.add_node("hitl_gate", hitl_gate_node)                 # 🟢 阈值判断
graph.add_node("publish", publish_node)                     # 🟢 纯代码
graph.add_node("human_review", human_review_node)           # 🔴 人工介入（非 Token）

# 条件边：无事件时跳过 event_encode
graph.add_conditional_edges(
    "build_evidence",
    lambda s: "event_encode" if s["evidence"]["has_unstructured_event"] else "score_difficulty"
)

# 条件边
graph.add_edge("event_encode", "score_difficulty")
graph.add_conditional_edges("quality_gate", 
    lambda s: "debate" if s["disagreement"]["requires_debate"] else "fuse")
graph.add_conditional_edges("hitl_gate",
    lambda s: "human_review" if s["hitl"]["required"] else "publish")
```

**注意事项**：
- 深拷贝 State 在状态膨胀时 Token 开销高 → 用 `TypedDict` + `Annotated` 限制状态键，只传必要数据
- LangGraph 默认串行 → 用 `Send` API 实现多个 Skill 并行执行
- Checkpoint 用 PostgreSQL 做生产持久化（不用 InMemorySaver）
- LangGraph 运行在 AgentScope 平台内部，不独立暴露服务端点

### 2.3 自研预测层：领域核心

**为什么必须自研？**AgentScope / LangGraph 是"舞台和演员"——负责 Agent 通信、编排、HITL——但它们不理解物流预测的领域知识（如"暴雨衰减因子 0.7""客户上报虚高率 15%""22 点后 ETA 集中时 30% 跨班"）。以下是必须自研的五个模块。

#### 2.3.1 Evidence Blackboard（统一事实来源）

**职责**：所有 Skill 读取数据的唯一入口。不存真实数据，只存**结构化引用**。

```python
# 数据结构
class EvidenceEntry:
    type: Literal["timeseries", "event_text", "vehicle_eta", "customer_notice", "human_note"]
    source: str          # 数据来源：weather_api, oe_system, kadm, im_message
    timestamp: datetime
    content_ref: str     # 指向实际数据的引用（S3 URI / DB query / API endpoint）
    ttl_seconds: int     # 缓存有效期
    quality_score: float # 数据质量分（完整度/新鲜度/一致性）
```

**证据类型枚举**（对应 12 份调研方案的数据输入）：

| 证据类型 | 来源 | 对应方案 |
|---------|------|---------|
| `timeseries_volume` | SATIS/OE 历史件量 | A1, A2, A3, B1, B2 |
| `business_zone_forecast` | 业务区人工预测 | A1 |
| `city_dynamic_forecast` | SATIS/OEWM 城市动态预测 | B1 |
| `vehicle_eta_list` | OE 陆运实时监控 | C2, H1 |
| `customer_reported_volume` | kadm 大客户摸底 | D1 |
| `weather_alert` | 气象 API | F1 |
| `diversion_record` | SATIS/OE 倒货记录 | G2 |
| `human_note` | 场地主管 IM/备注 | 全部方案的人工修正 |

#### 2.3.2 Event Encoder（非结构化事件→特征向量）

**职责**：整个预测链路中 **LLM Token 消耗的核心入口**。暴雨预警文本、客户临时通知、修路公告等非结构化事件，在进入 Skill 执行前统一由 Event Encoder 语义解析为结构化特征向量。

**为什么放在 Blackboard 之后、Skill 执行之前？**
- 事件编码只需执行 **一次**，结果被所有 Skill 共享复用
- 如果每个 Skill 各自调 LLM 理解事件，Token 消耗会放大多倍
- 编码后的结构化向量存入 Evidence Blackboard，Skill 只读结构化数据执行

```python
def event_encode_node(state: ForecastState):
    """将非结构化事件文本编码为结构化特征向量，结果写入 Evidence Blackboard。"""
    events = state["evidence"].get("unstructured_events", [])
    if not events:
        return {}  # 无事件时跳过，0 Token 消耗
    
    encoded = []
    for event in events:
        # 🔴 LLM 语义编码（~500-1500 tokens/次，一次编码，所有 Skill 复用）
        vector = llm_encode_event(event)
        encoded.append(vector)
    
    # 写入 Evidence Blackboard，供后续所有 Skill 读取
    state["evidence"]["event_vectors"] = encoded
    return {"evidence": state["evidence"]}
```

**事件编码输出示例**（即 Skill 实际消费的结构化输入）：

```json
{
  "event_type": "weather_extreme",
  "severity": 4,
  "affected_areas": ["shanghai"],
  "time_window": {"start": "14:00", "end": "20:00"},
  "estimated_volume_decay": 0.65,
  "estimated_delay_hours": 3,
  "confidence": 0.75
}
```

#### 2.3.4 Bayesian Fusion（多 Skill 融合）

**职责**：将多个 Skill 的预测分布融合为一个。每个 Skill 的历史可靠性（MAPE 衰减）作为精度权重。

```python
def bayesian_fusion(claims: List[ForecastClaim]) -> FusedForecast:
    """Bayesian precision-weighted fusion of multiple Skill claims."""
    for claim in claims:
        # 历史可靠性权重
        reliability = exp(-MAPE_i / tau) * coverage_i * evidence_quality_i
        # 精度 = 可靠性 / 方差
        precision_i = reliability / (claim.sigma ** 2)
    
    # 精度加权融合
    mu_fused = sum(precision_i * claim.mu) / sum(precision_i)
    sigma_fused = sqrt(1 / sum(precision_i) + sigma_residual ** 2)
    
    return FusedForecast(mu=mu_fused, sigma=sigma_fused)
```

#### 2.3.5 Conformal Calibration（置信区间校准）

**职责**：防止 Skill 过度自信。用历史残差分桶校准置信区间。

```python
def conformal_calibrate(fused: FusedForecast, bucket_key: str, alpha=0.2):
    """Conformal prediction: calibrate PI using historical residuals."""
    residuals = load_residuals(bucket_key)  # 按 site_type × event_type × horizon 分桶
    q = quantile(residuals, 1 - alpha)
    
    return CalibratedForecast(
        mean=fused.mu,
        pi_lower=fused.mu - q,
        pi_upper=fused.mu + q
    )
```

#### 2.3.6 Thompson Sampling 权重学习（自进化）

**职责**：每个 Skill 在每个场景 bucket 下维护 Bayesian 后验，上线时抽样决定选用哪些 Skill。

```python
def select_skills_with_thompson_sampling(
    bucket: str, 
    candidate_skills: List[str], 
    n_select: int
) -> List[str]:
    """Thompson Sampling for dynamic team selection."""
    scores = []
    for skill_id in candidate_skills:
        posterior = load_posterior(skill_id, bucket)
        # 从后验分布抽样
        sample = np.random.normal(posterior.mu, posterior.sigma)
        scores.append((skill_id, sample))
    
    # 选 top-N
    selected = sorted(scores, key=lambda x: x[1], reverse=True)[:n_select]
    return [s[0] for s in selected]


def update_posterior(skill_id: str, bucket: str, reward: float):
    """T+1 实际件量回流后更新后验."""
    posterior = load_posterior(skill_id, bucket)
    
    # 指数衰减 + 新观测
    posterior.mu = 0.92 * posterior.mu + 0.08 * reward
    posterior.sigma = max(0.05, posterior.sigma * 0.98)
    posterior.n += 1
    
    save_posterior(skill_id, bucket, posterior)
```

### 2.4 HITL 协作层：HiClaw-style

**选型理由**：只在高风险场景触发（分歧指数 DI ≥ 0.50 或业务影响高），让场地主管/区域经理在协作房间中旁观 Agent 主张、人机共同决策。

**触发条件**：

| 条件 | 动作 |
|------|------|
| DI < 0.20 | 直接融合发布 |
| 0.20 ≤ DI < 0.35 | 加权投票自动消解 |
| 0.35 ≤ DI < 0.50 | 1 轮 Peer Review |
| DI ≥ 0.50 | **触发 HiClaw-style 协作房间** |
| 任一高影响事件缺证据 | **触发 HITL** |

**协作房间 UI 元素**：

```text

│  高风险预测复核 — 021WD 金山场地               │

│  Agent 主张：                                │
│  ┌─────────────────────────────────────────┐ │
```

### 2.7 Token 成本设计原则

**核心法则**：LLM 只做语义理解和异常推理，数值计算、统计融合、比例缩放、查表运算全部走代码。

#### 2.7.1 全链路 Token 消耗分类

| 节点 | 消耗类型 | Token 估算 | 触发条件 |
|------|:---:|:---:|------|
| **意图解析 (parse_intent)** | 🔴 LLM | ~300 | 每次请求（固定） |
| **构建证据板 (build_evidence)** | 🟢 代码 | 0 | 每次请求 |
| **事件编码 (event_encode)** | 🔴 LLM | ~800 | 有非结构化事件时 |
| **难度评分 (score_difficulty)** | 🟢 代码 | 0 | 每次请求（VAE 小模型推理） |
| **动态选团队 (select_team)** | 🟢 代码 | 0 | 每次请求（Thompson Sampling） |
| **Skill 执行 (run_skills)** | 🟢 代码 | 0 | 12 个 Skill 中 10 个纯代码 |
| **质量门禁 (quality_gate)** | 🟢 代码 | 0 | 每次请求 |
| **有界辩论 (bounded_debate)** | 🔴 LLM | ~1500 | 仅 DI ≥ 0.35 时触发，max 2 轮 |
| **贝叶斯融合 (bayesian_fusion)** | 🟢 代码 | 0 | 每次请求 |
| **共形校准 (conformal_calibration)** | 🟢 代码 | 0 | 每次请求 |
| **HITL 门禁 (hitl_gate)** | 🟢 代码 | 0 | 每次请求（阈值比较） |
| **发布预测 (publish)** | 🟢 代码 | 0 | 每次请求 |

#### 2.7.2 12 个 Skill 的 Token 消耗分类

| 方案 | 计算方式 | Token | 原因 |
|------|:---:|:---:|------|
| **A1** 业务区占比拆解 | 🟢 | 0 | 业务区预测值 × 历史占比 → 纯乘法 |
| **A2** 班次占比拆分 | 🟢 | 0 | 日总量 × 白/晚班占比 → 纯乘法 |
| **A3** 时效推演拆解 | 🟢 | 0 | 按运输时效查表分桶 → 纯查表 |
| **B1** 城市动态涨跌幅 | 🟢 | 0 | 基期实际 × (城市预测/城市基期) → 纯算术 |
| **B2** 上周同比 | 🟢 | 0 | 上周同期 × 涨跌修正 → 纯算术 |
| **C2** 在途车辆推演 | 🟢 | 0 | 24h ETA 车辆 × 单车载重 → 纯运算 |
| **D1** 大客户摸底 | 🟡 | 少量 | 客户通知文本的语义理解依赖 event_encode（共享）；核心计算是代码 |
| **E1** 集散分拆合并 | 🟢 | 0 | 集货 + 散货分口径计算后加总 |
| **F1** 极端事件应对 | 🟡 | 少量 | 气象预警语义理解依赖 event_encode（共享）；区间评估是代码 |
| **G2** 倒货管理 | 🟢 | 0 | 倒货率查询 + 来源产能量化 → 纯运算 |
| **H1** 交叉验证 | 🟢 | 0 | 多数据源数值比对 → 纯比较逻辑 |
| **H2** 四维度加总 | 🟢 | 0 | 四个维度各自用代码 Skill 算完 → 加总 |

> **关键设计**：D1 和 F1 的 LLM 消耗不在 Skill 内部，而是在前置的 `event_encode` 节点统一完成。Skill 本身只消费已编码的结构化特征向量。

#### 2.7.3 单次预测 Token 消耗预估

```text
平峰日常（无事件 / 无分歧）：       ~300 tokens
有事件 + 无分歧：                 ~1100 tokens  (意图 + 事件编码)
有事件 + 高分歧 + 辩论（≤2轮）：  ~2600 tokens  (意图 + 事件编码 + 辩论)

对比基线：如果每个 Skill 都是 LLM Agent 自由对话
          → 单次预测轻松突破 20,000 tokens ❌
```

#### 2.7.4 离线 Token 消耗（不计入在线 SLA）

## 四、预测主流程伪代码

```python
def forecast(request: ForecastRequest) -> ForecastResponse:
    trace = new_trace(request)
    
    # === Step 1-3: 意图解析 + 取数 + 事件编码 + 难度评估 ===
    intent = parse_intent(request.text)                      # 🔴 LLM ~300 tokens
    evidence = build_blackboard(intent, trace)               # 🟢 纯代码
    evidence = event_encode(evidence)                        # 🔴 LLM ~800 tokens（有事件时）
    difficulty = score_difficulty(intent, evidence)          # 🟢 VAE 推理
    
    # === Step 4: 动态团队选择（Thompson Sampling） ===
    team = select_team_with_thompson_sampling(               # 🟢 纯概率计算
        intent=intent,
        evidence=evidence,
        difficulty=difficulty,
        registry=skill_registry,
    )
    
    # === Step 5: 并行执行 Skill ===
    claims = run_skills_parallel(                            # 🟢 10/12 个 Skill 0 Token
        team=team,
        evidence=evidence,
        timeout=budget_for(difficulty),
        trace=trace,
    )
    
    # === Step 6: 质量门禁 ===
    claims = quality_gate(claims)                            # 🟢 规则校验
    
    # === Step 7: 分歧检测 + 条件辩论 ===
    disagreement = calc_disagreement(claims, intent)         # 🟢 纯计算
    if disagreement.requires_debate:
        claims = bounded_debate(claims, evidence,             # 🔴 LLM ~1500（分歧时，max 2轮）
                                max_rounds=debate_rounds(difficulty))
    
    # === Step 8: Bayesian Fusion ===
    fused = bayesian_fusion(claims)                          # 🟢 纯数学
    
    # === Step 9: Conformal Calibration ===
    calibrated = conformal_calibrate(                         # 🟢 纯统计
        fused, bucket=scenario_bucket(intent, evidence))
    
    # === Step 10: HITL 门禁 ===
    hitl_decision = hitl_gate(calibrated, disagreement,       # 🟢 阈值比较
                               business_impact=intent.impact)
    
    if hitl_decision.required:
        return send_to_human_review(calibrated, claims, evidence, trace)
    
    # === Step 11: 发布 + 记录 ===
    publish_forecast(calibrated, trace)                      # 🟢 纯代码
    return build_response(calibrated, claims, trace)
```

## 六、关键技术风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| AgentScope 框架较重，团队学习曲线陡峭 | Phase 0 延期 | 预留 1-2 周学习期；先跑官方 tutorial；如遇阻塞可降级为纯 LangGraph 方案（已在 Phase 0 覆盖） |
| LangGraph State 深拷贝导致 Token 开销高 | Phase 1 成本超预期 | 使用 `TypedDict` 限制状态键；Prompt Caching 降 90% 输入成本；Semantic Caching 消除 ~31% 冗余 |
| Skill 冷启动时 Thompson Sampling 权重不稳定 | Phase 3 效果差 | 先验设置为保守值（小 sigma）；前 30 次采用均匀探索；Similar-Site Skill 迁移提供先验 |
| HITL 协作房间触发过于频繁 | 业务人员"AI 疲劳" | 严格限制触发阈值（仅 DI ≥ 0.50 或高业务影响）；监控每周触发率 |
| 12 个调研方案转化为 Skill 的准确度不达预期 | Phase 2 交付受阻 | 先转 3-5 个核心方案（B1/D1/F1/G2/H1）；回放不通过 → 人工修正 → 重新回放 |
